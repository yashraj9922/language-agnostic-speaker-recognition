────────────────────────────────────────────────────────────────────────────────
Q1.  How is the UI working when a user tests their own audio,
     and how are the .pt files used in the dashboard?
────────────────────────────────────────────────────────────────────────────────

PART A — Try It Yourself UI Flow
─────────────────────────────────

The UI has two phases:

  ENROLLMENT
  ──────────
  1. User records or uploads 1–5 audio clips for a speaker (e.g. "Alice").
  2. Each clip goes through:
       load_audio()          → decode bytes → resample to 16 kHz → mono tensor
       center_crop_or_pad()  → take center 2-second window (32,000 samples)
       encoder(wav)          → wav2vec2 → AttentivePool → Projection → L2-norm
                             → 256-dim embedding vector
  3. If multiple clips: mean of all embeddings is computed, then re-normalized.
  4. The final vector is stored in st.session_state.enrolled["Alice"].
     Nothing is written to disk — it lives in browser session memory only.
     Refresh = profiles gone.

  VERIFICATION
  ────────────
  1. User records or uploads a test clip.
  2. Same pipeline produces a 256-dim embedding for the test clip.
  3. Cosine similarity is computed against every enrolled speaker:
       score = dot(test_embedding, enrolled_embedding)
       (dot product equals cosine similarity when both vectors are L2-normalized)
  4. Scores are ranked. If the top score >= EER threshold (~0.54):
       ✅  Verified as [Name]
     Otherwise:
       ❌  Speaker Not Recognised


PART B — How the .pt File Is Used
───────────────────────────────────

  The checkpoint file  checkpoints/best_model.pt  contains:
    ┌─────────────────────────────────────────────┐
    │  model_state  →  all learned weight tensors │
    │  epoch        →  9                          │
    │  val_eer      →  8.25                       │
    │  epoch_log    →  list of 10 epoch dicts     │
    └─────────────────────────────────────────────┘

  Two separate uses in app.py:

  1. load_encoder()  [cached once at startup]
     - torch.load() reads the file
     - n_spk = state["loss_fn.W"].shape[0]  → 108 speakers
     - SpeakerModel is built with correct size, weights loaded
     - Only the encoder half is returned; AAMSoftmax head is discarded
     - Cached in RAM for the entire session via @st.cache_resource

  2. load_epoch_log()  [powers Training Dynamics page]
     - Reads epoch_log, best epoch, val_eer from the checkpoint
     - Falls back to HARDCODED_EPOCH_LOG if file is missing

  Flow diagram:
    best_model.pt
      ├── model_state ──→ load_encoder() ──→ embed() ──→ Try It Yourself
      └── epoch_log   ──→ load_epoch_log() ──→ Training Dynamics charts


────────────────────────────────────────────────────────────────────────────────
Q2.  What does the .pt file do specifically when a user
     tests their own audio on the UI?
────────────────────────────────────────────────────────────────────────────────

The .pt file is loaded ONCE at app startup — when a user records audio,
the .pt file plays zero role. Here is the exact sequence:

  AT APP START (one time only)
  ────────────────────────────
    best_model.pt
      → torch.load()
      → model_state (weights dictionary)
      → SpeakerModel built in Python (random weights initially)
      → model.load_state_dict(weights)   ← fills in the learned numbers
      → encoder extracted and cached in RAM
      → .pt file is never read again

  WHEN USER RECORDS AUDIO
  ────────────────────────
    Your audio clip
      → encoder (already in RAM from .pt)
      → wav2vec2 layers process the raw waveform
      → AttentivePool compresses frames → [2048]
      → Projection head → [256]   ← your voice fingerprint
      → L2 normalize → point on 256-dim hypersphere

  The encoder is just a math function at this point — a big matrix
  multiplication chain. The .pt file gave it the right numbers so
  those multiplications produce meaningful speaker embeddings
  instead of random noise.

  Simple analogy:
    .pt file         = instruction manual that programs a calculator
    load_encoder()   = programming the calculator once at startup
    User recording   = pressing buttons on the already-programmed calculator
    256-dim vector   = the number shown on the display

  The .pt weights encode what "same speaker" looks like in 256-dim space.
  Without them, cosine similarity scores would be ~0 for everyone,
  the threshold would never be crossed, and nothing would verify.


────────────────────────────────────────────────────────────────────────────────
Q3.  How did fine-tuning help in generating the L2-norm vector
     and the overall speaker verification pipeline?
────────────────────────────────────────────────────────────────────────────────

BEFORE FINE-TUNING
──────────────────
  wav2vec2 was pretrained on 56,000 hours of speech via contrastive
  self-supervised learning. It learned phoneme structure, pitch, rhythm,
  and acoustic texture — but had NO concept of speaker identity.
  Two different people saying the same sentence would score similarly
  because it encoded WHAT was said, not WHO said it.

WHAT FINE-TUNING ACTUALLY DOES
────────────────────────────────

  Step 1 — AAM-Softmax loss forces speaker separation on the hypersphere

    For each training batch:
      embedding → cosine similarity with each of 108 speaker prototypes (W)
               → add angular margin m=0.2 to the correct speaker's angle
               → multiply by scale s=30
               → CrossEntropy loss

    The margin means: your embedding must be 0.2 radians closer to your
    own prototype than standard softmax requires. This is a harder target.

      Without margin:  cos(θ_correct) must beat others
      With margin:     cos(θ_correct + 0.2) must beat others

    Effect: same-speaker embeddings cluster tightly; different-speaker
    embeddings are pushed apart — all on the unit hypersphere.

  Step 2 — Selective layer unfreezing

    Layers 1–20   ❄️ Frozen    → keep acoustic / phonetic knowledge
    Layers 21–24  🔥 Trainable → shift from "what is said" → "who is speaking"
    Pool + Proj   🔥 Trainable → learn to summarize speaker identity

    The bottom layers already encode rich speech features. The top 4 layers
    get redirected from phoneme representation to speaker-discriminative
    representation.

HOW THE L2-NORM VECTOR IS GENERATED (step by step)
────────────────────────────────────────────────────

  Raw waveform [32,000 samples]  (2 seconds × 16,000 Hz)
        ↓
  CNN Feature Extractor (frozen, stride=320)
        ↓  32000 / 320 = 100 frames
  [100, 1024]  — one 1024-dim vector per 20 ms frame
        ↓
  Transformer layers 1–20 (frozen)
        ↓
  Transformer layers 21–24 (fine-tuned)  ← encode "who", not "what"
        ↓  still [100, 1024]
  Attentive Statistics Pooling
        ↓
    α = softmax( Linear(1024→128→1) )    ← which frames matter most?
    μ = Σ αt · xt                         ← weighted mean  [1024]
    σ = sqrt( Σ αt · (xt - μ)² )         ← weighted std   [1024]
    out = concat(μ, σ)                    ← [2048]
        ↓
  Linear(2048→512) → BatchNorm → ReLU
  Linear(512→256)
        ↓
  L2 Normalize:  v = v / ||v||            ← project onto unit sphere
        ↓
  256-dim unit vector  (voice fingerprint)

WHY L2 NORMALIZATION IS THE KEY STEP
──────────────────────────────────────
  Without normalization, vector magnitude also carries information — noise.
  L2 norm forces every embedding onto a unit sphere (||v|| = 1 always),
  so ONLY DIRECTION matters.

  Cosine similarity between two unit vectors simplifies to:
    cos_sim(A, B) = dot(A, B)   (since ||A|| = ||B|| = 1)

  Same speaker    → embeddings point in same direction → score close to 1.0
  Different speaker → embeddings point apart           → score close to 0.0

  AAM-Softmax uses cos(θ) everywhere — it is designed for the unit hypersphere.
  The L2 norm is not an afterthought; training directly optimizes
  the DIRECTION of embeddings, so normalization is a requirement.

CONCRETE IMPROVEMENT FROM FINE-TUNING
──────────────────────────────────────
                          Before fine-tuning    After fine-tuning
  Same speaker score      ~0.4–0.6             ~0.71
  Different speaker score ~0.3–0.5             ~0.37
  Score separation (Δμ)  ~0.10                ~0.34
  Val EER                 ~45%                  8.25%  (epoch 9)

  The fine-tuning widened the genuine/impostor gap from ~0.1 to ~0.34.
  That gap is what makes thresholding meaningful.

ONE-LINE SUMMARY
────────────────
  Fine-tuning taught the top 4 Transformer layers to ignore the words
  and encode only voice characteristics; L2 normalization ensures that
  "voice similarity" is purely a question of angular distance on a
  256-dim sphere, which cosine similarity directly and efficiently measures.
